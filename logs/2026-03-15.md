# KELLYPHOTO 開發日誌 - 2026-03-15 (問題分析與解案擬定)

## 任務背景
- 開發者要求讀取 3/11 與 3/12 的日誌，並分析之前 `kellyphoto` 頁面無法顯示照片的卡關癥結。
- 當時的系統無法順利將網址 Slug (`kellyphoto`) 對應到後端的實體租戶 ID (`kelly`)。

## 分析結果與問題根源
- **代碼層面**：`[slug]/page.tsx` 中實作了「雙認證邏輯」：先透過 `where("slug", "==", slug)` 去查 `tenants` 集合，藉此帶出 `resolvedTenantId`。
- **資料層面狀態**：目前資料庫中 `tenants` 集合只有一部名為 `tenant_kelly_owner` 的文件，其內部的 slug 是 `kelly`，而**不存在** slug 為 `kellyphoto` 的租戶文件。
- **失敗結論**：前端用 `kellyphoto` 下去查，查不到任何 `resolvedTenantId`，最後導致套用的查詢條件變成 `tenantId in ["kellyphoto", "kellyphoto"]`，完全沒撈到 354 張標記為 `kelly` 的相片與分類。這就是前面一直「抓不到資料」的真正原因。

## 擬定解決方案 (已寫入 Implementation Plan)
已針對此「ID 對齊」問題提出了三個可行的作法：
1. **方案一：前端寫死映射 (Hardcode Mapping)**
   - 發現 slug 為 `kellyphoto` 時強制指派 `resolvedTenantId` = `kelly`。
   - 優點：非常快，不影響資料庫，風險最低。
2. **方案二：資料庫補齊關聯 (Database Association)**
   - 在 `tenants` 集合補上一筆文件，將 `slug="kellyphoto"` 指向這個空間。
   - 優點：完全符合雙重認證的原有架構，前端無需修改。
3. **方案三：全面資料遷移 (Data Migration)**
   - 把所有資料庫裡的 `kelly` 全部改成 `kellyphoto`。
   - 缺點：風險高，可能影響現存圖片路徑。

### 新增探討：方案四 (別名機制) 與最佳建議
- **分析：** 除了先前三種方案，考慮到平台長期擴充性，也評估了「後端/中介層別名機制 (Alias System)」。但此作法目前可能過度設計（Over-engineering）。
- **結論：** 強烈推薦採用**方案二**（直接修改 	enants 集合文件中的 slug 欄位），這符合既有雙重認證架構，且不需大改前端。
- **下一步：** 撰寫完成 implementation_plan.md，等待開發者決定方案後執行。

### 追蹤研究：新增照片機制的影響
- **發現：** 在 AutoForm 與 CameraUploadButton 中，前端都會將當前網址的 slug (即 kellyphoto) 作為 	enantId 送出到後端 /api/upload。
- **結論：** 若 ID 問題不解決，未來所有新上傳的照片，其 	enantId 都會錯誤地變成 kellyphoto，而與舊照片 kelly 產生嚴重的資料割裂。這確立了「方案二：全面修改 tenants 集合的 slug」是唯一正確的根本解決之道。
- **進度：** 已更新 Implementation Plan 等待確認。

### 解答開發者疑問：舊照片的標記為何？
- **分析：** 開發者詢問目前現存 354 張照片以及分類是如何被標記的。
- **結論：** 原有的 354 張照片與相關分類，因為是在之前的環境上傳的，它們其實是被標記為 kelly（也就是當初系統認定為正確的身分證 ID）。這也就是為何今日前端使用 kellyphoto 會查不到資料（也是今天一開始抓不到照片的原因）。
- **後續影響：** 所以只要我們現在實作了「傳遞真實身分證 ID (kelly)」 給子元件的機制，這些舊照片就能跟未來的新照片完美會合於 kelly 身分證之下，且雙查詢陣列也能完美撈出它們。

### 一、Props Drilling 修復（tenantId 傳遞機制）

**問題：** 多個子元件（AutoForm、CameraUploadButton、CategoryManager、WorkManager、SystemSettings）都在用 `useParams().slug` 直接當作 tenantId，導致送出的是網址名稱 `kellyphoto` 而非真正的身分證。

**修復：** 在 `[slug]/page.tsx` 中建立 `resolvedTenantId` 變數，並透過 Props 傳遞給所有五個子元件。每個子元件也同步修改為接受 `tenantId` prop。

**修改檔案清單：**
- `src/app/[slug]/page.tsx` — 傳遞 resolvedTenantId
- `src/components/AutoForm.tsx` — 接收 tenantId prop
- `src/components/CameraUploadButton.tsx` — 接收 tenantId prop
- `src/components/CategoryManager.tsx` — 接收 tenantId prop
- `src/components/WorkManager.tsx` — 接收 tenantId prop
- `src/components/SystemSettings.tsx` — 接收 tenantId prop

### 二、排版跑版修復

**問題：** 在做 Props Drilling 時，不小心把 `page.tsx` 裡面的 Grid 排版（CategoryManager + AutoForm 的 12 欄配置）、`<hr>` 分隔線、以及 `isWorkManagerEnabled` 展開/收合面板全部移除了，造成後台大跑版。

### 三、Slug 解析機制（核心修復）

**問題：** `resolvedTenantId = slug` 只是偷懶地把網址名稱當身分證，沒有做真正的解析。

**修復：** 在 `HomeContent` 加入獨立 `useEffect`，查詢 Firestore `tenants` 集合：
`
where("slug", "==", slug) → 取出 tenantId
`
所有 8 處 `where("tenantId", "==", slug)` 改為 `resolvedTenantId`，2 處依賴陣列同步修改。

**Build 結果：** `npm run build` 通過（Exit code: 0）。

### 四、根因確認（最重要的發現）

透過臨時腳本直接查詢 Firestore，得到以下真相：

**portfolio_items 集合：**
- 356 張照片的 `tenantId` = `"kelly"`

**tenants 集合：**
| 文件 ID | slug | tenantId |
|---|---|---|
| tenant_kelly_owner | kelly | tenant_kelly_owner |

**問題一：** tenants 集合裡**沒有** `slug: "kellyphoto"` 的文件。
- 進入 `/kellyphoto` → 解析失敗 → 用 `"kellyphoto"` 查照片 → 0 張

**問題二：** 唯一的 Kelly 文件（`slug: "kelly"`）指向 `tenantId: "tenant_kelly_owner"`，但照片上印的是 `"kelly"`。
- 進入 `/kelly` → 解析到 `"tenant_kelly_owner"` → 用 `"tenant_kelly_owner"` 查照片 → 0 張

**結論：** 照片的 `tenantId` 與 `tenants` 集合的 `tenantId` 對不上。

### 追查與修復 (2026-03-15 後續)

**發現真正的 Root Cause：**
- 發現所有 `kelly` 舊照片之所以會吃錯標籤，是因為在給 Kelly 開通管理員權限時，由 `src/app/api/admin/sync-role/route.ts` 這支 API 以及 `src/actions/admin.ts` 所負責。
- 這兩支程式在去 `global_tenants` 查詢擁有者身分時，**錯誤地提取了網址名稱 (`slug`) 取代真實的 `tenantId`**，並將它寫死進了 Clerk 的 `publicMetadata.appAccess.photo_slug` 欄位中。
- 導致後續 Kelly 登入後，上傳 API 強調安全防衛時，竟拿著這個錯的 slug 當作真理，強制覆寫給了每一張照片。

**修復方案執行：**
1. **修正源頭查詢**：將 `sync-role/route.ts` 跟 `actions/admin.ts` 內的函式重構為 `getPhotoTenantIdForUser`。它現在只會讀取並回傳最底層的真實 `tenantId`（例如 `tenant_kelly_owner`）或文件 `id`。
2. **正名與釐清**：將寫入 Clerk 雖然欄位依舊叫做 `photo_slug`，但內容已經換成了真實 ID。同時在 API 最終端 `/api/upload/route.ts` 跟 `auth-middleware.ts` 裡，把取出這個值的變數名稱從 `tenantSlug` 正名為 `tenantId`，防止未來開發者誤解。
3. **驗證與編譯**：排除因備份檔 `.bak.ts` 引起的 TypeScript 衝突，重新執行 `npm run build` 通過。

**等待開發者下一步驗證：**
1. 去 Clerk Dashboard 把 Kelly 帳號中的 metadata `photo_slug` 清空。
2. 重新觸發一次角色同步（或重新派發權限）。
3. 使用 Kelly 帳號於 `/kelly` 頁面上傳新照片，檢查資料庫中的 `tenantId` 是否終於被正確地打上 `"tenant_kelly_owner"`。
