# KELLYPHOTO 專案日誌 - 2026-03-26

## 任務背景
使用者針對 Firestore 與身分驗證架構提出詢問：應該採「分開專案」還是「總集合」架構？

## 討論與決策
根據 `GEMINI.md` 規範與 SaaS 微服務最佳實踐，建議採取以下混合模式：

1. **身分驗證 (Authentication)**: 採 **「總集合」** 模式。
   - 使用 **Clerk** 作為唯一的身份驗證中心。
   - 優點：實現 SSO (單一登入)，使用者只需一個帳號即可存取旗下所有系統（作品集、口袋預約、口袋照片）。
   - 租戶隔離：利用 Clerk Organizations 的 `orgId` 作為跨系統的 `tenantId`。

2. **資料庫類型 (Database Type)**: **NoSQL (Firestore)**。
   - 採非關聯式 (Non-relational) 文件模型 (Document database)。
   - **格式區分**：非 MySQL, SQL Server 或 PostgreSQL 等傳統 SQL 資料庫。
   - **優點**：架構靈活（Schema-less）、原生支援實時顯示、橫向與縱向擴充快速。

3. **專案隔離原則 (Project Isolation)**:
   - **口袋管理 (Pocket Admin)**：應擁有專屬的 **「全域管理 (Global/Admin)」** Firebase 專案。
   - **子程式 (Micro-apps)**：如「口袋相片」、「作品集」，每個租戶或每個產品應擁有獨立的 Firebase 專案。
   - **現狀檢查**：目前 `pocket-admin` 與 `portfolio` 均共用 `kellyportfolio-811ca`，建議未來進行遷移隔離，以符合 SaaS 安全規範。

4. **統一身分識別與管理流程 (Unified Identity Flow)**:
   - **來源統一**：全線產品共用同一個 Clerk Application 實例。
   - **權限核心**：利用 Clerk Organization (OrgId) 作為 `tenantId`。
   - **存取橋接**：子程式透過 `/api/auth/firebase-token` 將 Clerk 身份轉換為該專案專屬的 Firebase Token。
   - **管理入口**：不論從官網還是子程式進來，皆由 Clerk Middleware 強制導向統一登入頁。

5. **維安風險分析 (Security Risk Analysis)**:
   - **防禦機制一：JWT 簽名**。Token 內的角色資訊經加密簽章，前端或竊取者無法自行修改。
   - **防禦機制二：短期過期**。Session Token 在數分鐘內即會過期，降低被盜用的窗口時間。
   - **防禦機制三：遠端撤銷**。管理員可透過 Clerk 後台強制登出特定使用者或所有設備。
   - **防禦機制四：資料庫最後防線**。核心寫入操作除檢查 Token 外，仍會對比 Firestore 的專案權限，確保物理隔離。
